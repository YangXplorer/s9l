# goal-check —— 目标与标准的自审闸门

这是 s9l 项目的"完成判定"机制。规则只有一句：

> **目标(Goal) 与 标准(Standard) 两者都达成 = ゴール。在 ゴール 之前，凡有不满足之处，必须自己审查、列出、修复、再审，直到达成——不得草率宣布完成。**

Goal 与 Standard 的**权威定义在 `s9l-goalkeeper` agent 里**（`.claude/agents/s9l-goalkeeper.md`），本 skill 负责驱动审查闭环。

## 何时运行
- 准备把某个 `docs/TASKS.md` 任务标记为 `[x]` 之前
- 用户说"做完了 / 可以合了吗 / 这个阶段完成了"
- `commit` / `push` / 发版之前
- 自己感觉"差不多了"的时候——尤其是这种时候

## 执行步骤

1. **确定审查对象与范围**
   - 是哪个交付？（任务ID / PR / 阶段 / 一组改动）
   - 收集证据：`git status`、`git diff`、相关源文件、对应 `docs/TASKS.md` 条目与所属 Phase 验收标准。

2. **调用 `s9l-goalkeeper` 做审查**
   - 用 Agent 工具启动 `s9l-goalkeeper`，把审查对象、改动范围、相关上下文交给它。
   - 它会逐条核验 Goal 与 Standard，输出 Gap 清单与判定（ゴール達成 / 未達成）。

3. **按判定分流**
   - **🎯 ゴール達成**：向用户报告达成，附 goalkeeper 的核验摘要。可以标记任务完成 / 继续 commit。
   - **🚧 未達成**：**不要宣布完成**。
     a. 把 Gap 清单转成具体的待办（必要时用 TaskCreate 跟踪）。
     b. 逐项修复（自己改，或交给 `personal-engineer` 实现）。
     c. 修完后**回到第 2 步重新审查**（再次跑 goalkeeper）。
     d. 循环，直到判定为 ゴール達成。

4. **收尾**
   - 仅在 goalkeeper 判定 ゴール達成 后，才向用户确认"已达成目标与标准"，并简述本轮修复了哪些 gap。
   - 如果某条款确实无法满足（外部阻塞/需用户决策），不要假装达成——明确标注为阻塞项，列出待用户拍板的点，暂停在"未達成 + 已知阻塞"。

## 原则
- **基于事实，不基于声称**：审查要看真实 diff / 跑命令，不靠"我觉得做完了"。
- **闸门不放水**：只要有一条适用条款未满足，就是未達成，继续迭代。
- **不扩范围**：修 gap 时只补"已定标准要求但缺失"的部分，不顺手加计划外功能。
- **诚实暂停优于虚假达成**：遇到真阻塞，宁可停在"未達成 + 阻塞说明"，也不谎报 ゴール。
